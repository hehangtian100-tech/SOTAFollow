### 📌 第一步：定义锚点 (Anchors) 的数据结构

feature_dim = 128

### 🪟与🗄️ 第二步：构建滑动窗口与全局记忆库
为了让当前帧能够与过去交互，模型会在内存中维护两个池子。由于每个锚点自带坐标，我们直接把特征和坐标拼在一起存起来。

```python
# 1. 局部参考窗口 (Local Window): 存最近 N 帧的锚点
local_window_features = torch.randn(1000, feature_dim) # 最近的 1000 个锚点
local_window_coords = torch.randn(1000, 3)

# 2. 全局轨迹记忆库 (Global Memory): 存历史关键帧的锚点
global_memory_features = torch.randn(50000, feature_dim) # 积累的 50000 个历史锚点
global_memory_coords = torch.randn(50000, 3)

# 为了方便进行注意力计算，通常会把局部和全局的锚点合并为 "上下文 (Context)"
context_features = torch.cat([local_window_features, global_memory_features], dim=0)
context_coords = torch.cat([local_window_coords, global_memory_coords], dim=0)
# 现在 context_coords 的形状是 [51000, 3]

```

### 📐 第三步：核心创新——基于 3D 距离的掩码注意力 (Distance-Masked Attention)
这就是论文最亮眼的地方。标准的 Transformer 会让当前帧的 256 个点与上下文的 51000 个点进行密集的点乘计算。但这篇论文利用了 **3D 距离生成了一个掩码 (Mask)**，直接阻断了距离过远的注意力链接。

```python
class GeometricContextAttention(nn.Module):
    def __init__(self, feature_dim, search_radius):
        super().__init__()
        self.search_radius = search_radius # 设定物理空间的感知半径，比如 2.0 米
        # 标准的多头注意力机制
        self.attn = nn.MultiheadAttention(embed_dim=feature_dim, num_heads=8, batch_first=True)

    def forward(self, query_feat, query_coord, key_feat, key_coord):
        # 1. 计算当前点集和上下文点集之间的 3D 欧氏距离
        # query_coord: [256, 3], key_coord: [51000, 3]
        # cdist 会生成一个 [256, 51000] 的距离矩阵
        distances = torch.cdist(query_coord, key_coord) 

        # 2. 生成几何上下文掩码 (Mask)
        # 在 PyTorch 中，True 代表 "忽略 (屏蔽) 这个位置"
        # 只要距离大于设定的物理半径，就禁止它们之间产生信息交互
        geom_mask = distances > self.search_radius 

        # 3. 执行受限的注意力计算
        # query (当前帧) 去查阅 key/value (历史上下文)
        output_features, _ = self.attn(
            query=query_feat.unsqueeze(0), 
            key=key_feat.unsqueeze(0), 
            value=key_feat.unsqueeze(0), 
            attn_mask=geom_mask  # 核心：注入 3D 空间约束！
        )
        
        return output_features.squeeze(0)

# 初始化模块并运行
gca_module = GeometricContextAttention(feature_dim=128, search_radius=2.0)
updated_curr_features = gca_module(curr_features, curr_coords, context_features, context_coords)

```
通过这三步代码，你应该能看清它的底层运作方式：**它用物理空间的距离矩阵（distances）强制掐断了 Transformer 的全连接特性，将其变成了符合现实 3D 规律的局部注意力网络。**
在上面的代码中，我们使用了 `torch.cdist` 来计算当前帧锚点与记忆库中所有 51,000 个锚点之间的距离矩阵。如果随着相机一直移动，全局记忆库里的锚点数量暴增到了 100 万个（或者更多），每次都用矩阵乘法去计算所有点对的距离，会严重拖慢帧率。

区分这两者是理解系统为什么能保持轻量化且不丢失信息的关键。简单来说，局部锚点是**“按时间自动淘汰”的，而全局锚点是“按空间几何特征精挑细选”**的。
我们来看看具体的筛选逻辑：

1. **局部锚点 (Local Anchors) —— 时间驱动的“滑动窗口”**
这里的筛选最简单粗暴：它其实就是一个**先进先出 (FIFO)** 的队列。系统直接提取**最近连续的 N 帧**（比如最近的 10 帧）产生的所有锚点。一旦有最新的第 11 帧进来，最老的那一帧的所有锚点就会被踢出“局部窗口”。
  - **目的：** 确保当前视野内的几何平滑，处理短时间内连续的、微小的相机移动。
2. **全局锚点 (Global Anchors) —— 空间驱动的“关键帧 (Keyframes)”**
当老的一帧被踢出局部窗口时，它并不会直接被丢弃，而是要面临一次“选拔”。系统不会把所有的历史帧都存入全局记忆库（那样会撑爆内存），而是只挑选**关键帧 (Keyframe)**。挑选标准通常基于以下几点：只有满足这些条件，这一帧的锚点才会被“晋升”为全局锚点，永久存入轨迹记忆库中。如果只是拿着相机站在原地发呆，画面没怎么变，这些无用的重复帧在离开局部窗口后就会被直接丢弃，从而极大地节省了存储空间。
  - **平移距离 (Translation)：** 相对于上一个全局关键帧，相机在 3D 空间中是不是移动了足够远的距离（比如超过了 0.5 米）？
  - **旋转角度 (Rotation)：** 相机的朝向是不是发生了显著的变化（比如转弯超过了 15 度）？
  - **视觉信息变化：** 当前画面和上一个关键帧相比，是不是出现了大量新的结构或纹理？
