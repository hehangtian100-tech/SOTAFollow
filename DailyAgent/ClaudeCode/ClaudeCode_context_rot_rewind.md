## 一句话总结

Claude Code 团队正式承认超长上下文（1M tokens）的副作用——随着对话历史积累，模型性能会"腐烂"，并给出了 5 个从"全清空"到"全保留"的梯度应对策略，其中 **Rewind（回退）被 Thariq 认定为最推荐方案**。

## Context Rot（上下文腐烂）

### 核心问题

Claude Code 团队的 **Thariq** 发文正式承认：
- **1M 上下文不是越长越好**
- 他们为这个现象正式命名：**Context Rot**（上下文腐烂）

### 腐烂曲线

| 上下文使用率 | 模型状态 |
|-------------|---------|
| **< 25%** | 健康，性能正常 |
| **25% - 50%** | 开始退化，被旧内容带偏 |
| **> 50%** | 公认拐点，风险急剧上升 |
| **满上下文** | 模型被大量不相关信息淹没 |

### 根本原因

随着对话历史积累：
- **不相关的内容**逐渐占据上下文空间
- 模型注意力被分散，对真正重要的信息响应质量下降
- 早期对话中的"垃圾"（过时假设、失败尝试）持续消耗模型认知资源

---

## 5个应对选项：从"全清空"到"全保留"

原文给出的 5 个选项**不是并列菜单，而是一个光谱**——从完全不保留上下文到全部保留，中间有一个特殊的 subagent 模式。

### 选项详解

```
光谱位置：                      少带 ←————————→ 多带
           [1] /clear                       [5] continue
                    [2] /compact                   
                    [3] subagent（只返回结论）
                    [4] rewind ← Thariq 最推荐
```

| # | 选项 | 策略 | 适用场景 |
|---|------|------|---------|
| 1 | `/clear` | 清空对话，自己写 brief | 任务完全转向，从零开始 |
| 2 | `/compact` | 让模型自己压缩总结 | 保留方向，清掉细节 |
| 3 | `subagent` | 派子任务，只带回结论 | 并行处理，独立任务 |
| 4 | `rewind` | 双击 ESC 回退到某步 | **Thariq 最推荐** |
| 5 | `continue` | 直接继续，不做处理 | 上下文还很干净时 |

## Rewind 为什么是 Thariq 最推荐的方案

### 典型失败场景

```
1. 读5个文件（A、B、C、D、E）
   → 上下文：5个文件的完整内容 ✓

2. 试方案 A，失败
   → 上下文：5文件 + A的失败记录 ✗

3. 试方案 B，又失败
   → 上下文：5文件 + A失败 + B失败 ✗✗

4. 此时 context 已被"垃圾"塞满
   模型既记得 A/B 的失败（影响判断）
   还在5个文件的旧假设里打转
```

### Rewind 的优势

**回到读完文件那一刻，不带 A/B 的失败记录，告诉模型"直接走 C"**

- ✅ 保留了关键信息（5个文件的理解）
- ✅ 甩掉了失败垃圾（A/B 的错误路径）
- ✅ 模型轻装上阵，不被历史失败干扰

### /compact vs Rewind

| | `/compact` | `rewind` |
|--|-----------|----------|
| **保留内容** | 模型自己总结，可能遗漏细节 | 精确保留想保留的部分 |
| **清掉什么** | 模型决定，可能清错 | 自己决定，精确控制 |
| **适用场景** | 方向正确，只需压缩细节 | 试错后需要甩掉垃圾 |

## 对 Claude Code 使用者的实际建议

### 任务切换时
- **不要**无脑 continue，先问自己：上下文现在干净吗？
- 如果任务已完全转向 → `/clear` 重新写 brief

### 遇到死胡同时
- 试了 2-3 个方案都失败 → 这时 context 已经脏了
- **不要继续在脏上下文上试** → `rewind` 回退

### 多任务并行时
- 用 `subagent` 派独立任务，每个子任务干净独立
- 主上下文只接收子任务的结论

### 上下文超过 50% 时
- 主动 compact 或 rewind
- 不要等到模型开始"发疯"（被旧内容带偏）
